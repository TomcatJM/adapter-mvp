# 钉钉需求文档标准模版

> 用途：产品在钉钉文档中写需求时，优先按这份模版组织内容。Codex/AI 读取文档后，也必须先对齐这份模版，再进入 workflow、云效创建、流水线和关单流程。

## 1. 使用原则

- 一篇文档可以包含多个云效“需求”。
- 每个“需求”下面可以包含多个云效“任务”。
- 任务必须挂在当前需求下面，不能跨需求借用标题、描述、负责人或主要内容。
- `版本号` 是文档级字段，用于映射云效迭代。
- `需求描述` 进入云效需求描述。
- `主要内容` 进入云效任务描述。
- 负责人为空时保留为空，由 Adapter 创建云效任务时按项目默认负责人兜底。
- 如果文档结构和本模版差异较大，先沉淀新的模版样式或补充解析规则，不直接猜测执行。

## 2. 文档头部

```text
# <项目名或模块名> <版本号> 需求文档

项目名：<Adapter 项目名，例如 jdb-demo>
版本号：<云效迭代版本号，例如 V1.0.0>
来源链接：<钉钉文档链接，可为空，由 Adapter workflow 补充>
需求标识：<本次需求的 requirementKey，可为空>
```

字段说明：

| 字段 | 是否必填 | 用途 |
| --- | --- | --- |
| `项目名` | 建议必填 | 用于匹配 Adapter 项目配置、云效项目、Apifox 项目 |
| `版本号` | 必填 | 用于匹配云效迭代，匹配不到时应失败 |
| `来源链接` | 可选 | workflow 已有钉钉 URL 时可不写 |
| `需求标识` | 可选 | 用于人工检索和幂等辅助，不替代 workflowId |

## 3. 需求与任务结构

```text
## 需求一：<需求标题>

需求描述：
<写清楚这个需求本身要解决的问题、业务背景、目标结果。这里的内容进入云效需求描述。>

验收标准：
- <需求级验收点 1>
- <需求级验收点 2>

### 任务一：<任务标题>

负责人：<负责人姓名，可为空>

主要内容：
- <任务要做的具体事项 1>
- <任务要做的具体事项 2>
- <接口、字段、页面、状态、权限、数据规则等具体要求>

接口影响：
- <新增接口 / 修改接口 / 无接口影响>

测试要点：
- <任务级测试点 1>
- <任务级测试点 2>

### 任务二：<任务标题>

负责人：

主要内容：
- <任务内容>

## 需求二：<需求标题>

需求描述：
<需求二的描述，只能归属于需求二。>

### 任务一：<任务标题>

负责人：<负责人姓名，可为空>

主要内容：
- <任务内容，只能归属于需求二下的任务一>
```

## 4. 完整示例

```text
# jdb-demo V1.0.0 需求文档

项目名：jdb-demo
版本号：V1.0.0
来源链接：https://alidocs.dingtalk.com/i/nodes/xxx
需求标识：jdb-demo-v1-student-follow

## 需求一：学生资料补全

需求描述：
补齐学生基础资料录入能力，支持后续销售跟进和校区运营使用。

验收标准：
- 新增学生时姓名、手机号必填。
- 手机号格式错误时给出明确提示。

### 任务一：新增学生基础信息字段

负责人：张三

主要内容：
- 学生姓名必填。
- 手机号必填。
- 新增来源渠道字段。
- 保存后返回学生 ID。

接口影响：
- 修改学生保存接口。

测试要点：
- 缺少姓名时不能保存。
- 缺少手机号时不能保存。

### 任务二：学生列表展示来源渠道

负责人：

主要内容：
- 学生列表增加来源渠道列。
- 支持按来源渠道筛选。

接口影响：
- 修改学生列表接口。

## 需求二：跟进记录优化

需求描述：
优化销售跟进记录录入，方便跟踪学生转化过程。

验收标准：
- 跟进内容不能为空。
- 跟进记录保存后能在学生详情中展示。

### 任务一：新增跟进记录必填校验

负责人：李四

主要内容：
- 跟进类型必填。
- 跟进内容必填。
- 保存成功后写入操作日志。

接口影响：
- 修改跟进记录保存接口。
```

## 5. AI 解析前检查清单

AI/Codex 在进入后续 workflow 前，必须先检查：

| 检查项 | 通过标准 |
| --- | --- |
| 是否有文档级 `版本号` | 能明确映射云效迭代 |
| 是否能识别需求标题 | 每个需求标题都是明确边界 |
| 是否能识别任务标题 | 任务位于某个需求标题下面 |
| 需求描述归属是否明确 | 需求描述只进入当前需求 |
| 任务主要内容归属是否明确 | 主要内容只进入当前任务 |
| 负责人是否明确 | 明确则写入，空则保留空 |
| 是否存在跨需求引用 | 存在时先暂停确认，不猜 |

不通过时，先补充或沉淀新的模版样式，再继续执行。

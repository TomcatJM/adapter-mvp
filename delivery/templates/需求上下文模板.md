# 需求上下文模板

> 用途：AI / Codex 在进入需求确认、云效建单或 coding spec 前，统一整理 `knowledgeContext`。本模板只描述上下文产物，不替代钉钉需求文档标准模版。

## 1. 基本信息

```text
workflowId: <Adapter workflow ID>
projectKey: <Adapter project key>
projectName: <项目名称>
requirementTitle: <需求标题>
sourceUrl: <钉钉文档链接>
branchName: <目标分支，例如 develop>
commitId: <用于 CodeGraph 查询的 commit，可为空>
indexVersion: <CodeGraph 索引版本，可为空>
```

## 2. 业务知识

```json
{
  "businessKnowledge": {
    "summary": "",
    "businessRules": [],
    "userScenarios": [],
    "knownConstraints": [],
    "relatedDocuments": []
  }
}
```

填写口径：

- `summary`：用一小段话说明当前需求所在业务域。
- `businessRules`：只写和本需求相关的规则，不泛化成全系统说明。
- `userScenarios`：写真实业务场景或操作路径。
- `knownConstraints`：写权限、状态、数据边界、历史兼容等约束。
- `relatedDocuments`：写知识图谱或文档索引返回的可追溯引用。

## 3. 研发入口

```json
{
  "developerContext": {
    "entrypoints": [],
    "relatedApis": [],
    "relatedTables": [],
    "relatedJobs": [],
    "existingTests": []
  }
}
```

填写口径：

- `entrypoints`：Controller、Service、Mapper、前端入口等可落地位置。
- `relatedApis`：接口路径、方法、入参/出参影响。
- `relatedTables`：只列本需求可能读写的表。
- `relatedJobs`：定时任务、消息、流水线节点等异步入口。
- `existingTests`：可复用或必须扩展的测试。

## 4. CodeGraph 影响面

```json
{
  "codeImpact": {
    "query": "",
    "branchName": "",
    "commitId": "",
    "indexVersion": "",
    "directFiles": [],
    "callers": [],
    "callees": [],
    "riskNotes": []
  }
}
```

填写口径：

- `query`：本次影响面查询问题。
- `directFiles`：直接修改或最可能修改的文件。
- `callers` / `callees`：上游调用方和下游依赖。
- `riskNotes`：跨模块副作用、兼容风险、历史坑点。

## 5. AI 执行提示

```json
{
  "aiPlanHints": {
    "recommendedSteps": [],
    "mustNotDo": [],
    "openQuestions": [],
    "acceptanceChecks": []
  }
}
```

填写口径：

- `recommendedSteps`：建议编码步骤，控制在可执行粒度。
- `mustNotDo`：明确禁止的静默默认、跨需求归属、误关单等行为。
- `openQuestions`：必须问清楚或必须失败的事项。
- `acceptanceChecks`：需求完成后要验证的业务和技术检查。

## 7. 使用规则

- 业务知识来自知识图谱查询，不从代码影响面反推业务规则。
- CodeGraph 只描述代码影响面，不替代业务知识图谱。
- `knowledgeContext` 可以增量补充，但不能覆盖已有 requirement、coding result、pipeline 信息。
- 如果项目配置、知识图谱或 CodeGraph 索引缺失，必须明确标记缺失原因，不能写空上下文假装成功。

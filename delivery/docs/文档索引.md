## 3. 建议维护方式

- 产品文档格式变化时，先更新 `../templates/钉钉需求文档标准模版.md`，再调整解析和云效创建逻辑。
- AI 研发闭环上下文变化时，先更新 `../templates/需求上下文模板.md` 和 `../templates/coding-spec模板.md`，再调整 workflow 或 Codex 侧实现。
- 新能力先写进 `主流程落地方案` 或 `后续配置清单`，再改代码。
- 需要操作者直接上手的流程，优先写进 `使用手册`。
- 涉及表结构、状态机、绑定规则的改动，优先同步到 `架构设计`。
- 涉及部署、环境变量、密钥和脚本的改动，优先同步到 `部署与配置`。

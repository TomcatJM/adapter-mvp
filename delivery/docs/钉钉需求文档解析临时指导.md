# 钉钉需求文档解析临时指导

> 目的：给 AI / Codex 一个短期内必须遵守的处理口径，确保钉钉需求文档解析、workflow requirement 入库、云效需求/任务创建这几段不会跑偏。

## 1. 适用范围

当前仅适用于这条主链路：

```text
钉钉需求文档
  -> Adapter 读取文档
  -> AI 解析为结构化 requirement
  -> Adapter 创建云效需求
  -> Adapter 基于需求创建云效任务
  -> 后续 coding / pipeline / Apifox / 关单
```

这份文档只约束前半段：

- 钉钉文档怎么理解。
- 结构化 requirement 应该长什么样。
- 一个文档该拆成几个云效需求和任务。
- 标题、负责人、版本号怎么处理。

不在这里处理：

- 代码实现细节。
- 流水线回调细节。
- Apifox 导入细节。
- 云效关单细节。

## 2. 当前固定前提

当前钉钉文档格式已经不是“一篇文档 = 一个任务”，而是更接近：

```text
文档标题
版本号
  需求一
    任务一
      负责人
      主要内容
    任务二
      负责人
      主要内容
  需求二
    任务一
    任务二
```

已确认的新口径：

1. `版本号` 是文档级字段。
2. `需求一 / 需求二 / ...` 是云效“需求”层级，应先创建为云效需求。
3. 每个需求下面的任务条目，是挂在该需求下面的云效任务。
4. `负责人` 是任务级字段。
5. `主要内容` 后面的多行文本，直接进入任务描述。

## 3. 固定处理原则

后续 AI 按下面原则处理，不要自行发挥：

### 3.1 一篇文档可以拆成多个云效需求，每个需求下再拆任务

- 不允许再按“整篇文档创建一个云效任务”处理。
- 一个 `需求` 标题对应一个云效需求。
- 一个云效需求下面可以有多个云效任务。
- 最终创建顺序必须是：先创建需求，再基于需求创建任务。

### 3.2 版本号挂在文档级，不复制成独立任务

- `版本号` 存在 `workflow` 的 requirement/context 中。
- 创建云效需求和任务时，版本号必须用于云效迭代字段映射。
- 如果项目配置里已有固定 `sprint_id`，优先使用该迭代。
- 如果项目没有固定 `sprint_id`，Adapter personal token 链路会按版本号查询云效迭代；能唯一匹配时写入创建 payload 的 `sprint` 字段，匹配不到或多匹配时应失败并提示补配置。
- 不把版本号单独创建成任务。

推荐需求标题格式：

```text
[V1.0.0] 需求一
```

推荐任务标题格式：

```text
[V1.0.0][需求一] 创建跟进记录
```

### 3.3 需求是一级工作项，不只是任务归属

- `需求一`、`需求二` 要创建成云效需求。
- 后续云效任务必须挂到对应云效需求下面；personal token 链路创建子任务时用父需求内部 ID 写入 `parentId`。
- 后续云效标题、描述、回写钉钉文档时，都要能看出该任务属于哪个需求。
- 不允许只创建任务而丢掉需求层级。

### 3.4 负责人为空时不乱猜

- 文档里如果 `负责人` 为空，结构化结果里保留空值。
- 创建云效任务时，才允许走“项目默认负责人”兜底。
- 不能在解析阶段把空负责人直接替换成某个人名。

### 3.5 主要内容后面的文本直接作为任务描述

- `主要内容` 后面的逐行内容全部保留。
- 不要在解析阶段丢行、合并成一句空泛摘要。
- 任务描述里应保留原始层次和文字顺序。

## 4. 结构化 requirement 目标格式

AI 读取钉钉文档后，提交到 workflow 的 requirement，至少要包含下面这层结构：

```json
{
  "documentTitle": "校务3-1.adoc",
  "version": "V1.0.0",
  "sourceUrl": "https://alidocs.dingtalk.com/...",
  "demands": [
    {
      "demandIndex": 1,
      "title": "需求一",
      "description": "描述：1111111",
      "items": [
        {
          "itemIndex": 1,
          "title": "任务一",
          "parentDemandIndex": 1,
          "parentDemandTitle": "需求一",
          "ownerName": "",
          "contentLines": [
            "创建一条学生信息",
            "姓名必填",
            "手机号必填"
          ]
        },
        {
          "itemIndex": 2,
          "title": "创建跟进记录",
          "parentDemandIndex": 1,
          "parentDemandTitle": "需求一",
          "ownerName": "",
          "contentLines": [
            "新建跟进记录",
            "类型、内容必填"
          ]
        }
      ]
    }
  ]
}
```

最少字段要求：

- `documentTitle`
- `version`
- `sourceUrl`
- `demands[].title`
- `demands[].demandIndex`
- `demands[].description`
- `demands[].items[].itemIndex`
- `demands[].items[].title`
- `demands[].items[].parentDemandIndex`
- `demands[].items[].parentDemandTitle`
- `demands[].items[].ownerName`
- `demands[].items[].contentLines`

## 5. 云效需求和任务创建映射规则

云效工作项创建必须按下面规则映射：

### 5.1 映射粒度

- `demands[]` 的每一项，对应一个云效需求。
- `demands[].items[]` 的每一项，对应一个云效任务。
- 云效需求要带上 `description`，对应文档里 `需求` 下的 `描述`。
- 云效任务必须挂在对应云效需求下面；不能只通过 `parentDemandIndex` / `parentDemandTitle` 文本记录父需求。
- 一个 workflow 可以创建多个云效需求和多个云效任务。
- workflow 主表仍可保留“主需求/首任务/最近任务”等字段，但不能因此丢掉需求和任务明细。

### 5.2 标题映射

需求标题建议生成：

```text
[版本号] 需求标题
```

例如：

```text
[V1.0.0] 需求一
```

任务标题建议生成：

```text
[版本号][需求标题] 任务标题
```

例如：

```text
[V1.0.0][需求一] 创建跟进记录
```

如果版本号为空，则退化为：

```text
[需求一] 创建跟进记录
```

### 5.3 描述映射

云效需求描述建议至少包含：

- 文档标题
- 版本号
- 需求标题
- `description` 原文
- `描述：` 后面的原始文本
- 原始钉钉文档链接
- 该需求下的任务清单

云效任务描述建议至少包含：

- 文档标题
- 版本号
- 所属需求
- 负责人
- `主要内容` 后面的原始文本
- 原始钉钉文档链接

### 5.4 负责人映射

- 如果 `ownerName` 非空，优先按人员表匹配。
- 如果 `ownerName` 为空，走项目默认负责人。
- 如果 `ownerName` 非空但匹配不到人员表，应明确报错或记录待人工处理，不静默改派给别人。

## 6. 非规整格式容错规则

当前文档已经暴露出格式不完全规整的问题，后续解析时必须容错。

### 6.0 不完全依赖固定文字或固定换行

解析时不能完全锁死在某几个固定文字、固定标点或固定换行上。

处理原则：

- 优先按文档结构层级解析，例如 `heading-4` 更接近需求，`heading-5` 更接近任务。
- `需求`、`描述`、`负责人`、`主要内容` 等文字标签只是辅助信号，不是唯一依据。
- 允许中英文冒号、空格、空行、换行位置发生变化。
- 如果后续文档基本沿用当前模板，应优先兼容当前模板；但不能因为某个标签少了冒号、换了换行或多了空行就解析失败。
- 当结构层级和文字标签冲突时，先保留原文和解析置信度，不要静默丢弃内容。

### 6.1 标题不规整

例如：

- 第一项叫 `任务一`
- 第二项不写 `任务二`
- 而是直接出现 `创建跟进记录`

处理规则：

- 优先把最近的非空标题识别为任务标题。
- 不强依赖“任务一/任务二/任务三”这种连续命名。
- 如果某块没有显式“任务X”，但出现一个独立业务短句标题，也可视为任务标题。

### 6.2 负责人为空

处理规则：

- 允许空值。
- 不视为解析失败。
- 创建云效时再兜底。

### 6.3 主要内容标签书写不一致

例如可能出现：

- `主要内容`
- `主要内容：`

处理规则：

- 标签识别时忽略中英文冒号差异。
- 标签后面的连续文本行都归入 `contentLines`，直到下一个同级标题或下一个需求开始。

### 6.4 空行和缩进不稳定

处理规则：

- 不依赖固定缩进空格数。
- 允许空行存在。
- 以“标题语义 + 标签语义 + 相邻段落”组合判断归属。

## 7. 明确禁止的做法

后续 AI / 实现中，禁止以下做法：

1. 把整篇钉钉文档当成一个云效任务。
2. 只提取摘要，不保留 `主要内容` 明细。
3. 在解析阶段擅自把空负责人替换成默认负责人。
4. 丢掉 `需求一 / 需求二` 这种需求层级，只创建任务。
5. 只把 requirement 存成一段自然语言，不保存结构化层级。
6. 因为格式有一点不规整，就回退到人工复制粘贴建单。

## 8. 推荐实施顺序

后续实现建议按这个顺序推进：

1. 先调整 requirement JSON 结构，支持 `demands[].items[]`。
2. 再调整 workflow requirement 入库与读取逻辑。
3. 再调整云效创建逻辑，从“单任务创建”改成“先按 demands 创建需求，再按 items 创建任务”。
4. 再补需求和任务明细落库，记录每个 demand 对应的云效需求 ID / 展示 ID，以及每个 item 对应的云效任务 ID / 展示 ID。
5. 最后再做钉钉文档回写、提交说明模板联动、流水线按任务绑定。

## 9. 当前阶段的临时结论

在新的钉钉文档格式下，当前最稳妥的落地口径是：

```text
1 个钉钉文档
  -> 1 个 workflow
  -> N 个云效需求
  -> N * M 个任务条目
  -> N 个云效需求工作项 + N * M 个云效任务工作项
```

因此，后续 AI / Codex 处理该类文档时，默认必须先做“结构化拆分”，再做“先需求后任务”的批量建单，不能跳过这一步。

## 10. 总体计划表

| 顺序 | 计划做什么 | 现在已具备的能力 | 还需要补全的能力 |
| --- | --- | --- | --- |
| 1 | 读取钉钉文档并识别模板 | Adapter 已能读取 Alidocs，当前模板已能识别 `版本号`、`需求`、`描述`、`任务`、`负责人`、`主要内容` | 继续提升容错，别死绑定文字和换行，优先按结构层级解析 |
| 2 | 把文档解析成结构化 requirement | workflow 已支持结构化需求提交，当前口径已明确 `需求描述` 和 `任务描述` 的分界 | 把 requirement schema 扩展成 `需求[] -> 任务[]` 的稳定结构，并保留原文 |
| 3 | 先创建云效需求 | 云效侧已经打通工作项创建，项目映射、人员表、默认负责人已可用 | 补“需求”这一层的创建接口和父子关联字段，避免只建任务不建需求 |
| 4 | 再基于需求创建云效任务 | 云效任务创建已真实联调成功，支持展示 ID、负责人解析、项目映射 | 改成从需求批量创建任务，并把每个任务挂到对应需求下面 |
| 5 | 把任务描述和需求描述写对 | 当前已明确：`描述` 进需求描述，`主要内容` 进任务描述 | 落库和接口返回时都要保留这两个层级，不要压扁成一段摘要 |
| 6 | 提交 coding result 并接流水线 | workflow P0、流水线回调、成功/失败绑定、Apifox 自动导入都已跑通 | 继续增强失败恢复和自动修复请求，避免人工来回切 |
| 7 | Apifox 同步后关单 | `APIFOX_SYNCED -> YUNXIAO_TASK_CLOSED` 已真实闭环 | 多项目状态配置、恢复策略、异常兜底还要继续补 |

### 10.1 当前可直接复用的能力

- Adapter 已支持 Alidocs 读取。
- workflow 已支持结构化需求提交。
- 云效工作项创建已真实联调。
- 云效任务关闭和回写已真实闭环。
- Apifox 导入已跑通。
- 人员表、项目表、默认负责人、展示 ID 都已经有基础能力。

### 10.2 当前最需要补的能力

- 文档结构化解析协议的稳定化。
- `需求 -> 任务` 的两级建单模型。
- 需求和任务的父子关联保存。
- 任务描述和需求描述的严格分层。
- 多项目下的自动匹配和失败恢复。

## 11. 执行纪律

后续按这份指导文件推进时，默认遵守下面两条：

1. 只有在需要你做明确决策时才暂停。
2. 每完成一步，都先做代码 review 和测试，再进入下一步。

补充要求：

- 不要为了省时间跳过自检。
- 不要把“先能跑”当成完成。
- 如果某一步的结果影响后续建单、解析或关单，必须先确认再继续。

# Adapter MVP 使用手册

这份手册面向日常使用者和联调人员，目标只有一个：把一次需求从钉钉文档跑到云效关单。

## 1. 你要先知道的事

- `adapter-mvp` 不是单纯的远端执行器，它还维护 workflow 账本、云效任务、Apifox 导入和审计记录。
- 主入口是钉钉/Alidocs 链接，后续流程都围绕这个链接展开。
- 提交说明里可以写 `WORKFLOW_ID`，也可以写 `YUNXIAO_TASK_DISPLAY_ID`、`云效id`、`云效任务` 这类字段；Webhook 读取的是本次提交的 `commitMessage`，不是分支最新提交。

## 2. 最小闭环

推荐按这个顺序跑：

1. `POST /workflow/start` 创建 workflow。
2. `POST /workflow/{workflow_id}/advance` 把状态推进到 `DOC_READ`。
3. `POST /workflow/{workflow_id}/requirement` 写入结构化需求。
4. 再次调用 `POST /workflow/{workflow_id}/advance` 创建云效任务。
5. 代码提交并触发流水线。
6. 流水线成功后自动导入 Apifox。
7. 导入完成后自动关闭云效任务。

## 3. 常见接口

### 3.1 创建 workflow

```http
POST /workflow/start
```

请求体示例：

```json
{
  "dingtalkUrl": "https://alidocs.dingtalk.com/i/nodes/<nodeId>",
  "requirementKey": "REQ-001",
  "operator": "codex",
  "context": {
    "projectName": "jdb-demo"
  }
}
```

### 3.2 读取钉钉文档

```http
POST /adapter/dingtalk/read
```

这个接口由 Adapter 托管钉钉 OpenAPI 配置，适合把需求文档变成可解析数据。

### 3.3 提交需求

```http
POST /workflow/{workflow_id}/requirement
```

建议把文档标题、版本、需求分组、任务条目、负责人、主要内容、验收标准一次性整理进去。主要内容后面的文本，建议写入任务描述。

### 3.4 创建云效任务

```http
POST /workflow/{workflow_id}/advance
```

当 workflow 到 `REQUIREMENT_PARSED` 时，这一步会调用云效创建工作项。当前支持：

- 单任务创建
- 按需求拆分成多个任务
- 指定负责人
- 没有指定负责人时，使用项目默认负责人

### 3.5 提交编码结果

```http
POST /workflow/{workflow_id}/coding-result
```

建议至少带上：

- 分支名
- 提交 ID
- 合并请求 URL
- 测试结果摘要

### 3.6 查看状态和审计

```http
GET /workflow/{workflow_id}
GET /adapter/status/{task_id}
GET /adapter/audit/{task_id}
```

## 4. 推荐提交说明

```text
feat: 修复学生列表筛选

云效任务: VEGZ-1186
工作流ID: wf-27f5462a4ec047f1
```

这类写法对流水线回调最稳。`WORKFLOW_ID` 优先，其次是云效展示 ID。

## 5. 失败时先看什么

- 钉钉读不到：先看 `./部署与配置.md` 的钉钉配置。
- 云效任务创建失败：先看项目映射、负责人映射、云效账号配置。
- Apifox 没同步：先看流水线映射和 `APIFOX_*` 配置。
- 关单失败：先看云效 `done_status_id` 或 `close_transition_id`。

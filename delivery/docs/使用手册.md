# Adapter MVP 使用手册

这份手册面向日常使用者和联调人员，目标只有一个：把一次需求从钉钉文档跑到云效关单。

## 1. 你要先知道的事

- `adapter-mvp` 不是单纯的远端执行器，它还维护 workflow 账本、云效任务、Apifox 导入和审计记录。
- 主入口是钉钉/Alidocs 链接，后续流程都围绕这个链接展开。
- 产品编写需求时先参考 [钉钉需求文档标准模版](../templates/钉钉需求文档标准模版.md)；AI 解析前也要先确认文档结构能落到这个模版。
- 提交说明里可以写 `WORKFLOW_ID`，也可以写 `YUNXIAO_TASK_DISPLAY_ID`、`云效id`、`云效任务` 这类字段；Webhook 读取的是本次提交的 `commitMessage`，不是分支最新提交。

## 2. 最小闭环

推荐按这个顺序跑：

1. `POST /workflow/start` 创建 workflow。
2. `POST /workflow/{workflow_id}/advance` 把状态推进到 `DOC_READ`。
3. 按标准模版检查钉钉文档，确认 `版本号 -> 需求 -> 任务 -> 负责人 -> 主要内容` 层级清晰。
4. `POST /workflow/{workflow_id}/requirement` 写入结构化需求。
5. 再次调用 `POST /workflow/{workflow_id}/advance` 创建云效任务。
6. 代码提交并触发流水线。
7. 流水线成功后自动导入 Apifox。
8. 导入完成后自动关闭云效任务。

## 3. 常见接口

### 3.1 创建 workflow

```http
POST /workflow/start
```

请求体示例：

```json
{
  "dingtalkUrl": "https://alidocs.dingtalk.com/i/nodes/<nodeId>",
  "requirementKey": "REQ-001",
  "operator": "codex",
  "context": {
    "projectName": "jdb-demo"
  }
}
```

### 3.2 读取钉钉文档

```http
POST /adapter/dingtalk/read
```

这个接口由 Adapter 托管钉钉 OpenAPI 配置，适合把需求文档变成可解析数据。

### 3.3 提交需求

```http
POST /workflow/{workflow_id}/requirement
```

建议把文档标题、版本、需求分组、任务条目、负责人、主要内容、验收标准一次性整理进去。主要内容后面的文本，建议写入任务描述。

提交前先按 [钉钉需求文档标准模版](../templates/钉钉需求文档标准模版.md) 做一次人工或 AI 检查：

- `版本号` 能匹配云效迭代。
- 每个 `需求` 都能创建为云效需求。
- 每个 `任务` 都能挂到当前需求下面。
- `需求描述` 只进入当前需求描述。
- `主要内容` 只进入当前任务描述。
- 结构不清晰时先补模版或提问，不继续猜测。

### 3.4 创建云效任务

```http
POST /workflow/{workflow_id}/advance
```

当 workflow 到 `REQUIREMENT_PARSED` 时，这一步会调用云效创建工作项。当前支持：

- 单任务创建
- 按需求拆分成多个任务
- 指定负责人
- 没有指定负责人时，使用项目默认负责人；如果任务已经写了负责人但 DB 未匹配，直接失败提示补配置，不使用默认负责人兜底

### 3.5 提交编码结果

```http
POST /workflow/{workflow_id}/coding-result
```

建议至少带上：

- 分支名
- 提交 ID
- 合并请求 URL
- 测试结果摘要

### 3.6 查看状态和审计

```http
GET /workflow/{workflow_id}
GET /adapter/status/{task_id}
GET /adapter/audit/{task_id}
```

### 3.7 删除云效工作项

```http
DELETE /yunxiao/workitems
```

删除接口只用于受控清理，不参与正常交付闭环：

- 必须使用带 `yunxiao:delete` scope 的数据库 token；普通 `ADAPTER_API_TOKEN` 不能删除。
- 默认 `dryRun=true`，只返回删除计划，不调用云效。
- 只接受明确的 `workitemIds` 或已记录的 `workflowId`，不按标题、项目名、负责人做模糊删除。
- 通过 `workflowId` 删除时，默认只纳入任务；`includeDemands=true` 才会把需求也放进计划，顺序是先任务后需求。

Dry-run 示例：

```json
{
  "projectName": "01-校务系统",
  "workitemIds": ["AYRR-4072", "AYRR-4073"],
  "dryRun": true
}
```

## 4. 推荐提交说明

```text
feat: 修复学生列表筛选

云效任务: VEGZ-1186
```

多个云效任务可以写在同一行：

```text
云效任务: VEGZ-1186、VEGZ-1187、VEGZ-1188
```

`云效任务` 是关单白名单。Adapter 只关闭这里显式列出的真实子任务；如果提交说明没有 `云效任务`，即使流水线和 Apifox 同步成功，也不执行云效关单。

如果流水线没有带到提交说明，可以在手动推进关单时补传：

```json
{
  "operator": "codex",
  "closeTaskRefs": ["VEGZ-1186", "VEGZ-1187"]
}
```

## 5. 失败时先看什么

- 钉钉读不到：先看 `./部署与配置.md` 的钉钉配置。
- 云效任务创建失败：先看项目映射、负责人映射、云效账号配置。
- Apifox 没同步：先看流水线映射和 `APIFOX_*` 配置。
- 关单失败：先看云效 `done_status_id` 或 `close_transition_id`。

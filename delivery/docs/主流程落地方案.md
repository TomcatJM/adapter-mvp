# Adapter MVP 主流程落地方案

## 1. 目标

这份文档把总体规划拆成可执行方案，避免后续实现时只按临时想法补接口。

主链路固定为：

```text
钉钉需求文档
  -> 创建 workflow
  -> 读取钉钉文档
  -> Codex 解析需求
  -> 创建云效工作项
  -> Codex coding
  -> 提交 coding result
  -> 云效流水线回调
  -> 同步 API 到 Apifox
  -> 关闭/回写云效工作项
```

每一步必须明确：

- 推荐方案。
- 为什么用这个方案。
- 其他方案为什么不优先。
- 是否依赖外部系统。
- 如果依赖外部系统，先确认有没有官方接口、哪些接口、怎么调用。
- 对应实现文档和验收标准。

## 3. 外部系统接口确认原则

只要某一步需要调用外部系统，必须先完成接口确认，再写实现。

确认内容包括：

- 官方产品、版本、endpoint。
- 接口 Action 或路径。
- 鉴权方式。
- 必填入参。
- 关键返回字段。
- 幂等依据。
- 失败返回格式。
- 是否需要本企业项目的字段配置。

禁止直接在代码里猜外部字段。

## 5. 为什么不用云效 CLI 做主链路

CLI 可以保留为人工调试工具，但不作为 Adapter 主实现。

原因：

- CLI 依赖运行机器安装、登录态、PATH 和版本。
- CLI 输出结构不如 HTTP/OpenAPI 稳定。
- CLI 日志更容易泄露 token、cookie 或参数。
- CLI 难做统一超时、重试、幂等和审计。
- CLI 难和 `adapter_workflow_event` 精准绑定。

主链路用服务端 HTTP/OpenAPI，才能统一落到 workflow 状态机。

## 8. 验收标准

方案级验收：

- 每个主流程步骤都有明确推荐方案。
- 每个外部系统步骤都有官方接口依据。
- 每个 P1 步骤都有对应任务文档。
- 不再出现“先写代码再猜接口”的路径。

实现级验收：

- 云效创建成功后 workflow 能查询到内部 `yunxiaoTaskId` 和页面展示 ID `yunxiaoTaskDisplayId`。
- 重复创建不会生成第二个云效工作项。
- 云效关闭只允许发生在 `APIFOX_SYNCED` 之后。
- 关闭成功后 workflow 状态为 `YUNXIAO_TASK_CLOSED`。
- 所有事件和错误都不包含 AK、token、cookie、Authorization、密码或私钥。

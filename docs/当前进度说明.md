## 1. 当前定位

`adapter-mvp` 目前已经不是单纯的 SSH 小工具，而是一个正在成型的交付集成服务。

当前更准确的定位是：

```text
Adapter MVP = 工具适配层 + 审计状态层 + 云效/钉钉/Apifox 集成入口
```

目标形态是：

```text
Adapter MVP = Workflow Orchestrator + Tool Adapter
```

也就是：Codex 负责理解需求和写代码，`adapter-mvp` 负责读钉钉、接云效、同步 Apifox、记录状态和审计。

### 2.2 鉴权和安全门禁

已完成：

- 业务接口使用 `Authorization: Bearer <ADAPTER_API_TOKEN>`。
- `/health` 不需要 token。
- `/adapter/execute` 必须带 `approvalId` 或 `approved=true`。
- 云效 execute 回调同样需要审批参数。
- 审计和响应中避免输出密码、token、私钥。

当前安全策略是对的：默认 preview，execute 必须显式审批。

### 2.3 Adapter Preview / Execute / Status / Audit

已完成统一 Adapter API：

```http

POST /adapter/execute
GET  /adapter/status/{task_id}
GET  /adapter/audit/{task_id}
```

当前已支持动作：

```text

```

用途：

- preview：只预览动作，不执行真实远端操作。
- execute：执行动作，但必须经过审批。
- status：查询执行结果。
- audit：查询审计事件。

### 2.6 Workflow P0 账本

已完成 workflow P0：

```http
POST /workflow/start
GET  /workflow/{workflow_id}
POST /workflow/{workflow_id}/advance
POST /workflow/{workflow_id}/requirement
POST /workflow/{workflow_id}/coding-result
```

当前能力：

- `/workflow/start` 从钉钉/Alidocs URL 创建 workflow 实例，状态为 `CREATED`。
- `/workflow/{id}/advance` 在 `CREATED` 状态读取钉钉文档，并推进到 `DOC_READ`。
- 钉钉读取失败会写 `doc_read_failed` 事件，状态保持 `CREATED`，便于修复配置后重试。
- `/workflow/{id}/requirement` 保存 Codex 结构化需求，并推进到 `REQUIREMENT_PARSED`。
- `/workflow/{id}/coding-result` 保存 branch、commit、MR、测试结果，并推进到 `CODE_SUBMITTED`。
- `GET /workflow/{id}` 返回实例和最近 workflow 事件。

这表示 P0 已经从“设计”变成“已落地的基础账本”。后续重点不再是建账本，而是把云效工作项、流水线和 Apifox/关单尾段接到账本上。

### Step 6：实现云效工作项创建

目标：结构化需求确认后，自动创建云效任务。

实现：

- `POST /workflow/{id}/advance` 在 `REQUIREMENT_PARSED` 状态调用云效 `CreateWorkitemV2`。
- 成功后写入 `yunxiao_task_id`，记录 `yunxiao_workitem_created` 事件。
- 随后记录 `coding_requested` 事件，并把状态推进到 `CODING_REQUESTED`。
- 如果 workflow 已有 `yunxiaoTaskId`，重复 advance 不会重复创建。
- 云效账号鉴权维护在 `adapter_yunxiao_account_config`：正式主链路支持 `personal_token` 和 `acs_ak`；`legacy_token` 只保留历史兼容创建路径；多个业务项目维护在 `adapter_yunxiao_project_config`。
- MySQL 已配置时，按 `projectName` / `affectedRepos` / `repoUrl` 解析项目并查表；缺少映射不会静默使用默认项目。
- 如果缺少云效账号鉴权、组织、项目、工作项类型或负责人配置，状态保持 `REQUIREMENT_PARSED`，记录 `yunxiao_workitem_create_failed`。
- 负责人解析顺序：`requirement.assigneeId/assigneeName` -> `adapter_yunxiao_project_member.is_default=1` -> 旧 `adapter_yunxiao_project_config.default_assignee` 兜底；指定负责人找不到时直接失败，避免错派。

已验证：

- 真实云效项目已创建工作项 `c24225e632abe9124f96a5811f`。
- workflowId、钉钉链接和需求描述已可写入云效任务。

### Step 7：Apifox 成功后关闭云效任务

目标：完整闭环。

实现：

- `POST /workflow/{id}/advance` 在 `APIFOX_SYNCED` 状态调用云效 HTTP/OpenAPI。
- 先查询工作项状态；如果云效侧已关闭，记录 `yunxiao_workitem_close_skipped` 并补偿推进。
- 未关闭时写入 Adapter 交付评论，再用 `done_status_id` 更新状态，或配置 `close_transition_id` 后走流转关闭。
- 成功后记录 `yunxiao_workitem_closed`，状态推进到 `YUNXIAO_TASK_CLOSED`。
- 失败时记录 `yunxiao_workitem_close_failed`，状态推进到 `NEEDS_HUMAN`，不误报成功。

已验证：

- 当前 `done_status_id=100014` 已用于真实关单。
- 个人令牌链路已完成回写和关单闭环。

待增强：

- 失败后的 `/workflow/{id}/resolve` 恢复策略。
- 对不同项目补齐 `done_status_id` / `close_transition_id` 配置。

## 7. 当前结论

`adapter-mvp` 现在已经完成了“工具底座 + workflow P0 账本 + 流水线成功到 Apifox 同步”的主链路：

- 能被云效调用。
- 能读钉钉文档。
- 能从钉钉 URL 创建 workflow。
- 能把钉钉读取结果、结构化需求、coding 结果写入 workflow。
- 能接流水线事件。
- 能分析失败。
- 能在流水线成功后同步 Apifox，并通过 workflow advance 完成云效关单。
- 能记录审计和状态。

但它还不是完整的交付 Workflow Orchestrator。当前最关键的下一步，不再是云效首尾两段能不能跑通，而是让真实流水线回调更稳、更可恢复：

```text
失败恢复和自动重试
真实流水线变量规范
```

账本已经有了，接下来要让外部系统按账本推进。
